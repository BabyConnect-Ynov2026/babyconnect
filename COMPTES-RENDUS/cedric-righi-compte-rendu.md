# Présentation projet

Le projet BabyConnect est une application web qui sert à simplifier l’utilisation des baby.
L’objectif est de faire un outil pour centraliser plusieurs tâches : voir les tables libres, gérer les
réservations,... De mon point de vue, le projet répond à un problème que j’ai déjà rencontré : le
fait d’aller jusqu’au souk pour voir s’il y avait un baby de libre.

# Rôle dans le projet

Dans l’équipe, j’ai travaillé à la fois sur fu frontend et du backend. J’ai pris en charge plusieurs
tâches importantes comme perfectionner l’authentification, faire la page accueil...

# Aides

J’ai travaillé avec mes coéquipiers, que ce soit en cas de question sur la technique ou même pour
les aider ave leur travail, j’ai par exemple aidé Ioané à régler des conflits de son code lorsque’il a
voulu push son travail.

# Contexte du bac à sable

Le contexte du bac à sable géant permet de tester de A à Z, sans avoir peur de toucher à des
vraies données d’utilisateur. C’est utile pour tester tout ce qui touche aux données en bdd, à la
gestion des réservations, au suivi live des tables.
Ce challenge m’a aussi permis de mélanger plusieurs domaines de compétences. Je n’ai pas
travaillé seulement en “frontend” ou seulement en “backend” : j’ai dû faire le lien entre interface,
API et logique métier.

# IA

J’ai utilisé l’IA de temps en temps sur ce qui concerne les tâches répétitives, par exemple pour le
style, pour obtenir des styles cohérents entre les pages.

# Shop

Personnellement je n’ai pas utilisé le shop.

# Difficultés et facilités

La principale difficulté était de bien déléguer les tâches, en étant si nombreux, aussi le fait de
devoir gérer toutes les couches (front et back) en même temps, un changement simple dans le
schéma ou dans une route pouvait casser le frontend, le seed. J’ai aussi rencontré des problèmes
de conflits, de ports, de proxy et de routes partiellement cassées après certaines modifications.

# Investissement personnel

Au-delà du code pur, je me suis investi dans le projet en essayant d’être utile sur ce qui faisait
vraiment avancer l’équipe : corriger les blocages, rendre les fonctionnalités testables, garder une
base exploitable, et éviter qu’on perde du temps sur des erreurs déjà identifiées. J’ai aussi pris le
temps de vérifier ce que je faisais, de relancer les builds, de tester les flux et de garder une trace
de ce qui avait été modifié.

Je considère que mon investissement ne se résume pas à une quantité de lignes de code. Il se
voit aussi dans ma capacité à reprendre des points cassés, à reconnecter des morceaux du projet
qui n’étaient plus alignés, et à fournir quelque chose de plus stable pour le reste de l’équipe.
